## 1. 선택 UX

### 1.1 콘텐츠 목록 — 체크박스 컬럼

```
┌────┬────────────────────────────────────┬──────┬──────┬──────────┐
│ ☐  │ 제목                               │ CP   │ 상태 │ 액션      │
├────┼────────────────────────────────────┼──────┼──────┼──────────┤
│ ☐  │ 기생충 (2019)                       │ CJ   │검수  │[상세]...  │
│ ☑  │ 부산행 (2016)                       │ Next │검수  │[상세]...  │
│ ☑  │ 미나리 (2020)                       │ A24  │검수  │[상세]...  │
│ ☑  │ 헤어질 결심 (2022)                  │ Moho │검수  │[상세]...  │
│ ☐  │ 오징어 게임 시즌1 (2021)            │ Net  │승인  │[상세]...  │
└────┴────────────────────────────────────┴──────┴──────┴──────────┘
```

### 1.2 헤더 체크박스 — 3-state

| 상태 | 의미 |
|------|------|
| `☐` | 아무것도 선택 안 됨 |
| `☑` | 현재 페이지 모두 선택 |
| `▣` (mixed) | 현재 페이지 일부 선택 |

헤더 클릭 → 현재 페이지 N개 토글. **전체 필터 결과**(다른 페이지 포함) 선택은 별도 confirm 배너로 분리 (§1.4).

### 1.3 선택 상태 표시 — 행 강조 + 상단 sticky 바

선택된 행: 좌측 파란 세로 막대 + 옅은 파란 배경.

선택된 콘텐츠가 1개 이상이면 **상단 sticky 바** 출현:

```
┌───────────────────────────────────────────────────────────────────────┐
│ [☑] 3개 선택  ·  현재 페이지 3/20  ·  [전체 필터 132개 선택]         │
│ [✓ 승인] [✗ 반려] [↻ AI 재처리] [🔍 외부 재매칭] [⋯ 더보기]  [✗해제]│
└───────────────────────────────────────────────────────────────────────┘
```

→ 액션 버튼은 **선택된 행들의 status 조합** 에 따라 동적 활성/비활성 (§2).

### 1.4 페이지 전반 선택 (Cross-page)

현재 페이지의 모든 항목을 선택했을 때, 상단에 작은 안내 배너:

```
┌───────────────────────────────────────────────────────────────────────┐
│ ℹ 이 페이지의 3개를 선택했습니다.                                     │
│   [→ 현재 필터의 132개 전체 선택]   [닫기]                            │
└───────────────────────────────────────────────────────────────────────┘
```

전체 선택 후:

```
┌───────────────────────────────────────────────────────────────────────┐
│ ☑ 필터 결과 132개 전체 선택됨.                                        │
│   [✗ 전체 해제]                                                       │
└───────────────────────────────────────────────────────────────────────┘
```

→ Cross-page 선택은 **명시적 confirm** 필요. 실수로 1000건 일괄 승인 방지.

### 1.5 키보드 단축키

| 키 | 동작 |
|----|------|
| `Shift+클릭` | 행 범위 선택 (마지막 클릭 ~ 현재 클릭) |
| `Ctrl/⌘+A` | 현재 페이지 전체 선택 |
| `Esc` | 선택 해제 |
| `↑/↓` 화살표 | 키보드 포커스 이동 |
| `Space` | 포커스된 행 토글 |

→ 검수자 파워유저(키보드 위주)를 위한 단축키. 모달 안에서는 비활성.

## 2. 액션 호환 매트릭스

`02_menu_lifecycle.md` §4.3 의 status 호환 규칙을 UI 액션으로 풀어쓴 표.

### 2.1 액션 7종

| 액션 | 호환 status | 결과 status | API |
|------|-----------|-----------|-----|
| ✓ 승인 | `staging`, `review` | `approved` | `POST /metadata/staging/bulk-approve` |
| ✗ 반려 | `staging`, `review` | `rejected` | `POST /metadata/staging/bulk-reject` |
| ↻ AI 재처리 | `review`, `processing.error` | `processing` | `POST /metadata/bulk/reprocess` (신규) |
| 🔍 외부 재매칭 | `review`, `staging` | (status 유지) | `POST /metadata/bulk/enrich` (신규) |
| ▶ 즉시 처리 | `waiting` | `processing` | `POST /metadata/bulk/process` (신규) |
| ↶ 회수 | `approved`, `rejected` (관리자) | `review` | `POST /metadata/bulk/recall` (신규) |
| 🗑 삭제 | (관리자) any | (DB row 삭제) | `DELETE /metadata/bulk` (신규) |

### 2.2 활성화 규칙

선택된 행들의 status 집합 S = {s1, s2, …}.

```
각 액션 A 의 호환 status 집합 = C(A)

A 활성: S ⊆ C(A)          (모든 선택이 호환)
A 비활성: S ⊄ C(A)         (호환 안 되는 status 1개라도 포함)
```

**예시**:

| 선택 조합 | 활성 액션 |
|----------|---------|
| `staging` × 3 | 승인, 반려, 외부 재매칭, 삭제(관리자) |
| `staging` × 2 + `review` × 1 | 승인, 반려, 외부 재매칭 |
| `review` × 5 | 승인, 반려, AI 재처리, 외부 재매칭 |
| `staging` × 2 + `approved` × 1 | (공통 없음) → 액션 모두 비활성, "다른 상태가 섞여 있습니다" 안내 |
| `processing.error` × 4 | AI 재처리 (실패 재시도 의미) |

### 2.3 비활성 버튼 hover 툴팁

비활성 버튼에 hover 시 사유 표시:

```
┌──────────────────────────────────────┐
│ ✓ 승인  (비활성)                      │
│                                      │
│ 선택한 콘텐츠 중 다음 상태는 승인    │
│ 불가:                                │
│   • 승인됨 (1개)                     │
│   • 처리 중 (2개)                    │
│                                      │
│ 호환되는 상태: staging, review        │
└──────────────────────────────────────┘
```

→ 운영자가 "왜 안 되는지" 즉시 파악. 호환 status 만 선택하도록 유도.

### 2.4 부분 적용 모드 (선택적)

회색 비활성 대신 **"호환되는 12개에만 적용"** 옵션도 제공:

```
┌──────────────────────────────────────┐
│ ✓ 승인하기                            │
│                                      │
│ 선택한 15개 중 12개만 승인 가능합니다│
│ (3개는 이미 승인됨/처리 중)          │
│                                      │
│ ◯ 호환되는 12개에만 승인              │
│ ◯ 취소 (전체 호환되도록 다시 선택)   │
│                                      │
│        [취소]    [12개 승인]         │
└──────────────────────────────────────┘
```

→ 운영자가 일일이 선택 해제하지 않아도 됨. 비호환 항목은 자동 skip + 결과 로그에 명시.

## 3. 액션 실행 흐름

### 3.1 동기 vs 비동기 분기

| 액션 | 처리 방식 | 이유 |
|------|---------|------|
| 승인 / 반려 (≤100건) | **동기** | DB 업데이트만. 빠름 |
| 승인 / 반려 (>100건) | **비동기 작업** | Celery batch job |
| AI 재처리 | **비동기** (항상) | 1건당 수초~수분, rate limit |
| 외부 재매칭 | **비동기** (항상) | TMDB/KOBIS API 호출 |
| 즉시 처리 | **비동기** | 큐 enqueue 동작 |
| 회수 / 삭제 | **동기** | 빠름 |

→ 임계값 100건은 mock. 백엔드 확인 후 ADR 로 추출.

### 3.2 동기 액션 — 즉시 결과

```
[✓ 승인] 클릭
    ↓
확인 모달 (§3.3)
    ↓
POST /metadata/staging/bulk-approve  body: {ids: [1, 2, 3]}
    ↓
응답: { approved: 3, failed: 0, results: [...] }
    ↓
토스트: "✓ 3개 콘텐츠를 승인했습니다.  [↶ 되돌리기]"
    ↓
목록 자동 갱신 (승인된 행은 status 칩이 초록색 "승인" 으로 전환)
```

### 3.3 확인 모달 (모든 액션 공통)

```
┌──────────────────────────────────────────────────────────┐
│  ⚠ 3개 콘텐츠를 승인하시겠습니까?                  [×]   │
├──────────────────────────────────────────────────────────┤
│  대상:                                                   │
│    • 부산행 (2016)         · 검수  → 승인                │
│    • 미나리 (2020)         · 검수  → 승인                │
│    • 헤어질 결심 (2022)    · 검수  → 승인                │
│                                                          │
│  사유 (선택):                                            │
│  ┌────────────────────────────────────────────────────┐ │
│  │                                                    │ │
│  └────────────────────────────────────────────────────┘ │
│                                                          │
│  ☐ 이 액션을 다시 묻지 않음 (10건 이하만)               │
│                                                          │
│              [취소]    [✓ 3개 승인]                     │
└──────────────────────────────────────────────────────────┘
```

→ **반려·삭제·회수**: 사유 입력 **필수** (감사 로그). 승인은 선택.

→ "다시 묻지 않음" 은 **소량 배치(≤10건)** 에만 허용. 그 이상은 항상 묻기 (실수 방지).

### 3.4 비동기 액션 — 진행 패널

```
[↻ AI 재처리] 클릭
    ↓
확인 모달 (사유/예상 비용 표시 — AI 호출은 비용 큼)
    ↓
POST /metadata/bulk/reprocess  body: {ids: [...]}
    ↓
응답: { job_id: "bj_8a3f...", total: 15, status: "queued" }
    ↓
상단 sticky 진행 패널 (이후 페이지 이동해도 유지)
```

상단 sticky 진행 패널:

```
┌───────────────────────────────────────────────────────────────────────┐
│ ⏳ AI 재처리 진행 중 — 7/15  ████████░░░░░░░  47%   [작업 상세]      │
└───────────────────────────────────────────────────────────────────────┘
```

→ 완료 시:

```
┌───────────────────────────────────────────────────────────────────────┐
│ ✓ AI 재처리 완료 — 성공 13, 실패 2.  [결과 보기]              [×]    │
└───────────────────────────────────────────────────────────────────────┘
```

[작업 상세] / [결과 보기] 클릭 → §3.5 작업 상세 페이지.

### 3.5 작업 상세 페이지 — `/programming/contents/jobs/{job_id}`

```
┌───────────────────────────────────────────────────────────────────────┐
│ AI 재처리 작업 #bj_8a3f...                                   [목록]   │
├───────────────────────────────────────────────────────────────────────┤
│ 시작: 2026-05-13 09:15   완료: 2026-05-13 09:22  소요: 7분          │
│ 트리거: 김PD             대상: 15개                                  │
│ 결과: ✓ 성공 13  ✗ 실패 2                                            │
│                                                                       │
│ ┌────┬──────────────┬────────┬─────────────────────────────────────┐ │
│ │ #  │ 콘텐츠       │ 결과   │ 메시지                              │ │
│ ├────┼──────────────┼────────┼─────────────────────────────────────┤ │
│ │ 1  │ 기생충 #1247 │ ✓ 성공 │ score 87 → 92                       │ │
│ │ 2  │ 부산행 #1248 │ ✓ 성공 │ score 71 → 84                       │ │
│ │ … │                                                              │ │
│ │ 14 │ 미나리 #1261 │ ✗ 실패 │ Gemini API: 429 Rate limit          │ │
│ │ 15 │ 헤어질 #1262 │ ✗ 실패 │ Ollama: connection refused          │ │
│ └────┴──────────────┴────────┴─────────────────────────────────────┘ │
```

→ **[↻ 실패만 재실행]** 핵심: 부분 실패 후 호출 가능. 검수자가 일일이 골라낼 필요 없음.

## 4. 부분 실패 처리

### 4.1 동기 액션의 부분 실패

```
POST /metadata/staging/bulk-approve  body: {ids: [1, 2, 3]}
응답: {
  approved: 2,
  failed: 1,
  results: [
    {id: 1, status: "approved"},
    {id: 2, status: "approved"},
    {id: 3, status: "error", error: "concurrent modification"}
  ]
}
```

토스트 (실패 있을 때):

```
⚠ 3개 중 2개 승인. 1개 실패.  [실패 항목 보기]  [↶ 되돌리기]
```

### 4.2 비동기 — 실패 행 강조

작업 상세 페이지에서 실패 행은 빨강 배경 + [↻ 개별 재시도] 버튼.

### 4.3 실패 사유 분류

| 카테고리 | 예 | UX 처리 |
|---------|-----|--------|
| Rate limit / 일시 | `429`, `503`, `connection refused` | [↻ 재시도] 권장. 자동 retry 안 함 (rate limit 가속 방지) |
| 데이터 충돌 | `concurrent modification` | 새로고침 후 재시도 |
| 권한 부족 | `403` | 관리자 문의 안내 |
| 입력 검증 | `invalid status transition` | 운영자 실수 — 다시 선택하라 안내 |

→ 백엔드는 `error_code` 필드를 분류 가능한 enum 으로 반환 (별도 task).

## 5. Undo (되돌리기)

### 5.1 Undo 가능 액션

| 액션 | Undo 가능 | 시간 제한 |
|------|---------|---------|
| 승인 | ✓ (관리자 회수) | 24시간 (그 후 명시적 회수만 가능) |
| 반려 | ✓ (회수) | 24시간 |
| AI 재처리 | ✗ (이전 결과는 ai_results 에 남아 [채택] 가능) | — |
| 외부 재매칭 | ✗ (이전 매칭은 external_meta_sources history 로 보존) | — |
| 회수 | ✓ (다시 승인/반려) | — |
| 삭제 | ✓ (soft delete) | 7일 |

### 5.2 토스트 [↶ 되돌리기]

승인/반려 직후 토스트에 [↶ 되돌리기] (24시간 내 클릭 가능):

```
✓ 3개 콘텐츠를 승인했습니다.  [↶ 되돌리기 (5)]
                                          ↑ 5초 자동 카운트다운 (이후도 24h 가능)
```

→ 짧은 카운트다운으로 **즉시 후회** 케이스 해결. 24시간은 늦은 후회.

### 5.3 Undo API

`POST /metadata/bulk/undo` body: `{action_id: "act_...", reason: "..."}` — 운영자 액션 로그(별도 task) 의 action_id 기반.

## 6. 필터 연동

### 6.1 필터 적용 후 다중 선택

현재 필터 결과 132건 → "전체 선택" → 132건 모두 선택됨 (cross-page 포함).

페이지네이션:
- 현재 페이지(20개) 보이는 행: 체크박스로 토글 가능
- 다음 페이지 이동 시: 선택 상태 유지 (sticky 바 카운트 유지)
- "필터 변경" 시: 선택 자동 해제 + 안내 ("필터를 변경하면 선택이 해제됩니다 [확인][취소]")

### 6.2 필터-액션 단축 패턴 — "검수 필요 + staging" 일괄 승인

```
┌────────────────────────────────────────────────────────────────┐
│ 필터: 검수 필요 ▾   세부: ☑ staging ☐ review                    │
│ 결과: 45개                                                      │
│ [☑ 45개 전체 선택]                                              │
│   → 활성 액션: [✓ 일괄 승인] [✗ 일괄 반려] [🔍 외부 재매칭]    │
└────────────────────────────────────────────────────────────────┘
```

→ 운영자 멘탈모델: "스테이징 큐 비우기" 가 가장 흔한 작업. 필터 → 전체 선택 → 일괄 승인 3클릭.

### 6.3 저장된 필터 + Bulk 액션

(미래 기능 — `dev-saved-views` 별도 task)

저장된 필터 "내 검수 큐 (staging, CP=Netflix)" + "이 필터의 전체 일괄 승인" 단축.

## 7. 운영자 액션 로그 (감사)

모든 Bulk 액션은 별도 테이블 `content_action_log` 에 기록 (별도 task — `dev-action-audit`):

| 컬럼 | 의미 |
|------|------|
| `id` | 액션 ID (Undo 시 참조) |
| `action_type` | approve/reject/reprocess/enrich/recall/delete |
| `actor` | 운영자 ID |
| `target_ids` | 대상 content_id 배열 |
| `reason` | 사유 (반려/삭제 필수) |
| `result` | success/partial/failed |
| `executed_at` | 실행 시각 |
| `undone_at` | Undo 된 경우 |

→ `04_content_detail.md` §7.3 의 변경 이력 사이드 패널이 이 테이블을 읽어 표시.
