## 2. 콘텐츠 라이프사이클

### 2.1 핵심 원칙 — **백엔드 enum 6개, UI 그룹 4개**

운영자에게 7~8개 상태를 노출하면 학습 비용이 커진다. **운영자 액션 관점**에서 보면 결국 4개 그룹으로 묶인다:

| UI 그룹 | 라벨 | 백엔드 enum | 운영자 액션 |
|---------|------|------------|-----------|
| 1 | 처리 중 | `waiting`, `processing` | 없음 (시스템 자동) |
| 2 | 검수 필요 | `staging`, `review` | 승인/반려/수정/재처리 |
| 3 | 승인됨 | `approved` | (옵션) 회수 |
| 4 | 반려됨 | `rejected` | (옵션) 회수 |

> 백엔드 enum 은 6개 그대로 유지 — DB 마이그레이션 없음. UI 필터/표시만 4개 그룹으로 묶음.

### 2.2 상태 전이도

```
                  [입력 진입점]
        단일 입력 / CSV 배치 / 외부 검색
                       │
                       ▼
            ┌───────────────────────────────┐
            │   UI 그룹 ①  "처리 중"        │
            │                               │
            │  ┌─────────┐    ┌──────────┐  │
            │  │ waiting │ ─▶ │processing│  │
            │  └─────────┘    └────┬─────┘  │
            └────────────────────┬─┴────────┘
                                 │
                  자동: AI 처리 완료 → score 산출
                                 │
                ┌────────────────┴────────────────┐
                ▼                                 ▼
   ┌─────────────────────────┐         ┌─────────────────────┐
   │  UI 그룹 ②  "검수 필요"  │         │ processing(error)   │
   │                         │         │  error_message      │
   │  ┌─────────┐ ┌────────┐ │         │  retry: ≤ 3         │
   │  │staging  │ │ review │ │         │                     │
   │  │ ≥ 90    │ │ < 90   │ │         │  운영자: 재시도/수동│
   │  └────┬────┘ └────┬───┘ │         └──────────┬──────────┘
   │       │           │     │                    │
   └───────┼───────────┼─────┘    재시도 → processing 로 복귀
           │           │                          │
   운영자 액션 (승인/반려/수정)                     │
           │           │                          │
           ▼           ▼                          │
   ┌────────────────┐ ┌────────────────┐         │
   │ UI 그룹 ③      │ │ UI 그룹 ④      │         │
   │ "승인됨"       │ │ "반려됨"       │         │
   │ ┌────────────┐ │ │ ┌────────────┐ │         │
   │ │ approved   │ │ │ │ rejected   │ │         │
   │ └────────────┘ │ │ └────────────┘ │         │
   └────────┬───────┘ └────────┬───────┘         │
            │                  │                  │
            └─── 회수 (관리자) ┘──────────────────┘
                  → review 로 복귀
```

### 2.3 백엔드 6개 enum 의 의미

| enum | 의미 | UI 그룹 | 시각 표시 (목록 행 배지) |
|------|------|--------|---------------------|
| `waiting` | 입력 완료, AI 처리 대기 | ① 처리 중 | 회색 "대기" |
| `processing` | AI 엔진 호출 중 (정상) | ① 처리 중 | 파란색 + 스피너 "처리 중" |
| `processing` (error) | AI 호출 실패, 재시도 대기 (`error_message` not null) | ① 처리 중 (실패 칩) | 빨간색 "실패" + 재시도 횟수 |
| `staging` | score ≥ 90, 자동 승인 후보 | ② 검수 필요 | 노란색 "자동검토" |
| `review` | 70 ≤ score < 90, 명시적 검수 필요 | ② 검수 필요 | 주황색 "검수" |
| `approved` | 최종 승인 | ③ 승인됨 | 초록색 "승인" |
| `rejected` | 반려됨 (사유 기재) | ④ 반려됨 | 회색 "반려" |

**중요**: `failed` 라는 별도 enum 은 추가하지 않음. 실패 정보는 `processing.error_message` + `retry_count` 필드로 표현. UI 에서는 "처리 중" 그룹 안에서 "실패" 칩으로 강조.

### 2.4 score 컷오프 (≥90 / 70~89)

- **≥ 90 (staging)**: AI 출력 자체로 자동 승인 후보. 운영자는 "이상 없음" 확인만.
- **70 ~ 89 (review)**: 필드별 확인 필요. 검수 시간 길어짐.
- **< 70**: AI 가 자신 없음. 외부 소스 매칭 강화 또는 운영자 수동 입력 필요. (현재는 `review` 로 분류되어 검수 큐로 떨어짐)

컷오프 값은 매직넘버 — `service.py` 상수 또는 환경변수로 추출 권장 (별도 task).

## 3. 콘텐츠 목록의 status 필터 (UI 4 그룹)

### 3.1 필터 칩 (상단)

```
┌────────────────────────────────────────────────────────────────┐
│  콘텐츠 목록                              [+ 콘텐츠 추가 ▾]    │
├────────────────────────────────────────────────────────────────┤
│ [전체 1,243]  [처리 중 15●]  [검수 필요 132◉]                  │
│ [승인됨 1,089] [반려됨 7]                                       │
├────────────────────────────────────────────────────────────────┤
│ 검색: [_______________] CP: [전체 ▾] 타입: [전체 ▾]            │
└────────────────────────────────────────────────────────────────┘
```

- `●` 표시: 시스템 알림 (예: 실패 N건 포함)
- `◉` 표시: 운영자 액션 대기 (검수 필요 — 강조)
- **default 선택**: "검수 필요" (역할이 검수자일 경우) 또는 "전체"

### 3.2 세부 카운트 (각 칩 내부 또는 hover)

"검수 필요 132" 칩 안에:
- 자동검토 가능 (staging, ≥90): 45
- 검수 필요 (review, 70~89): 87
- 점수 미달: (review 의 일부)

→ 풍부한 정보는 **칩 내부 mini-pill** 또는 **hover 툴팁** 으로 표시.

### 3.3 행 액션 (행 우측 버튼)

| UI 그룹 | 행 액션 |
|---------|--------|
| 처리 중 (waiting) | [상세] [즉시 처리] |
| 처리 중 (processing) | [상세] |
| 처리 중 (실패) | [상세] [↻ 재시도] |
| 검수 필요 (staging) | [상세] [✓ 승인] [✗ 반려] |
| 검수 필요 (review) | [상세] [✓ 승인] [✏ 수정] [↻ 재처리] |
| 승인됨 | [상세] [↶ 회수] |
| 반려됨 | [상세] [↶ 회수] |

## 4. 자동/수동 전이 정책

### 4.1 자동 전이 (시스템)

| From → To | 트리거 | 조건 |
|-----------|--------|------|
| `waiting → processing` | Celery worker | 큐 가용 + rate limit OK |
| `processing → staging` | AI 처리 완료 | score ≥ 90 |
| `processing → review` | AI 처리 완료 | score < 90 |
| `processing.error → processing` | 재시도 큐 (자동) | retry < 3 |

### 4.2 수동 전이 (운영자)

| From → To | UI 액션 | 권한 |
|-----------|--------|------|
| `staging → approved` | 행 [✓ 승인] / 상세 / Bulk 승인 | 검수자 |
| `staging → rejected` | 행 [✗ 반려] / 상세 | 검수자 |
| `review → approved` | 상세 검수 완료 + [승인] | 검수자 |
| `review → rejected` | 상세 [반려] (사유 기재) | 검수자 |
| `review → processing` | 상세 [AI 재처리] | 검수자 |
| `processing.error → processing` | 행 [재시도] (자동 3회 후) | 검수자 |
| `approved → review` | 상세 [회수] | 관리자 |
| `rejected → review` | 상세 [회수] | 관리자 |

### 4.3 Bulk 전이

목록에서 다중 선택 시 가능한 액션 (선택된 행들의 status 가 호환될 때만 활성화):

- `staging` 만 선택 N개 → 일괄 승인 / 일괄 반려
- `review` 만 선택 N개 → 일괄 AI 재처리 / 일괄 외부소스 재매칭
- `processing.error` 만 선택 N개 → 일괄 재시도
- 다른 status 가 섞이면 공통 액션만 활성화 (예: 어떤 status 든 "상세 보기" 만 가능)

→ `05_bulk_action.md` 참조.

## 5. 백엔드 변경 사항 (요약)

### 5.1 enum / 모델

- ContentStatus enum: **변경 없음** (waiting/processing/staging/review/approved/rejected 6개 유지)
- ContentMetadata: `error_message`, `retry_count` 필드 확인 — 없으면 추가 (별도 task)

### 5.2 score 컷오프

현재 매직넘버 70/90. ADR 또는 환경변수로 추출 권장.

### 5.3 의문점 (구현 시 결정 필요)

| # | 항목 | 결정 |
|---|------|------|
| 1 | `staging` 의 의미 — score≥90 자동검토 vs 외부매칭 완료 검토 | **score ≥ 90 + 외부 매칭 완료** 둘 다 충족 시 staging |
| 2 | score < 70 처리 | 현재 `review` 흡수 → "재처리 권장" 라벨 추가 |
| 3 | 재시도 횟수 | 3회 (현재 mock 값 — 백엔드 확인 후 ADR) |
| 4 | 권한 분리 (검수자/관리자) | RBAC 도입 별도 task |
